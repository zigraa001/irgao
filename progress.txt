## Codebase Patterns
- JS bundle is auto-generated from js/app/*.js via `npm run build:js` (scripts/build-app-js.js); never edit js/app.bundle.js directly
- Build script auto-bumps the ?v= cache-buster on CSS/JS links in app.html based on content hash
- For CSS-only changes (no JS edits), manually bump the ?v= on the css/app.css link in app.html
- Typecheck: `npm run typecheck` (requires `npm install` first if node_modules missing)
- Frontend is plain vanilla JS with ~75 top-level let/const globals shared across features in a single concatenated scope
- NEVER rename or remove element ids in app.html — js/app/*.js references them directly
- NEVER rename a CSS class without grepping js/app/*.js for it first (many classes are toggled/queried by JS); adding new classes alongside is safe
- Much JS-generated markup (ride cards, popular routes, admin tables/stat cards, toasts, dispatch cards) has its class names defined inside js/app/*.js — grep the relevant module before styling
- Watch for smart/curly quotes (' ') in JS — they pass typecheck but crash at runtime; always use ASCII quotes
- Google sign-in is live in production — keep .btn-google and all Google auth code intact
- Marketing pages (index.html, vision.html, team.html, contact.html) use css/tokens.css + other stylesheets — the app redesign must only touch css/app.css and app.html
- css/app.css is ~4000 lines organized by view sections with banner comments — edit in place, keep section organization
- Auth card visibility managed by hideAllAuthCards() with a hardcoded id array
- auth-routes.js manually re-registers routes from mountRoleSignup — when changing role-signup.js handlers, also update auth-routes.js
- New CSS tokens available: --focus-ring, --surface, --surface-subtle, --border-subtle, --transition — use these in future stories instead of hardcoding values
- app.html only loads css/app.css (not css/tokens.css) — brand tokens like --navy must be defined in app.css's :root if needed
- Service tab text labels are wrapped in `<span>` elements — use `.service-tab span { display: none; }` for icon-only responsive mode
- Role badges (.customer-role-badge, .op-role-badge, .admin-role-badge) are hidden at <=768px via media query
- The global :focus-visible rule in the RESET section applies to button, input, select, textarea, a — specific components can override with their own focus styles
- Location inputs are a connected container (.location-inputs) — inner .loc-input elements are borderless; the container itself holds border/radius/:focus-within
- For card-language consistency: use var(--surface) bg, var(--border-subtle) border, var(--radius-lg) corners, var(--shadow-sm) or var(--shadow-md) elevation
- dashboard-pill already has hover + .active state CSS — no need to add those when using it in new contexts
- Admin tables use `<div class="admin-table-wrap"><table class="admin-table">` pattern — auto-styled with uppercase headers, subtle dividers, row hover, rounded wrapper
- `.pd-input` is the standard input class for admin/drone forms (44px, rounded, focus ring) — use instead of inline styles
- `.admin-list-row` + `.admin-list-name` + `.admin-list-meta` for any list-of-items rows in admin sections
- Settings toggle JS sets inline `knob.style.left` and `track.style.background` — CSS provides base layout but JS overrides at runtime
- `.auth-input` and `.auth-btn` are now proper CSS classes for standalone form inputs/buttons outside .auth-field context
- WCAG AA contrast: never use var(--gray-400) for readable text labels — use var(--gray-500) minimum; gray-400 is OK only for placeholders/icons/decorative
- Touch targets on mobile: set min-height: 40-44px on interactive elements at ≤768px via media query
- Spacing rhythm: use 4px multiples (4, 8, 12, 16, 20, 24, 32, 40, 48) for padding/margin/gap

- **Learnings for future iterations:**
  - WCAG AA requires 4.5:1 contrast for normal text; var(--gray-400) (#94A3B8) on white is only 3.6:1 — always use var(--gray-500) (#64748B, 5.5:1) or darker for readable text; gray-400 is OK for placeholders, decorative elements, and icons
  - Multiple @media blocks at the same breakpoint cascade correctly — no need to merge them; later ones override earlier for same-specificity selectors
  - Touch target minimum for mobile (WCAG 2.5.8): 44px is ideal, 40px is acceptable — set min-height/min-width on buttons and interactive chips at ≤768px
  - 4px grid rhythm: stick to multiples of 4 (4, 8, 12, 16, 20, 24, 28, 32, 36, 40, 48) for padding/margin/gap — values like 10, 14, 18, 22 create visual inconsistency
  - The prefers-reduced-motion block should list ALL animated elements; the `* { transition-duration: 0.01ms !important }` catch-all handles transitions but explicit animation: none is needed for each @keyframes user
  - Manual browser verification was not possible — recommend visual QA at 360px, 768px, 1024px, and 1440px
