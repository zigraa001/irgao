## Codebase Patterns
- Stack: Express 5 + Prisma 6 (CommonJS, plain JS — no TS source). Prisma is PINNED to v6; do NOT upgrade to v7 (it removed `url = env(...)` from the datasource and requires driver adapters + prisma.config.ts, which breaks the provider-swap design).
- ENUMS: Prisma `enum` types are NOT supported on SQLite. Since the schema must run on BOTH SQLite (local) and MySQL (Hostinger), all "enum" columns (User.role, Aircraft.status, Booking.status) are plain `String`. Allowed values are documented in schema comments and MUST be enforced in application code (no DB-level constraint). Avoid DB-specific column types for the same cross-DB reason.
- MIGRATIONS: under the provider swap, generated migration SQL is dialect-specific, so prisma/migrations/ is NOT committed. Apply the schema with `npm run db:push` (works on either engine; schema.prisma is the source of truth). Do NOT rely on committed migrations.
- SEED: `npm run db:seed` (prisma/seed.js) is idempotent — users upsert by unique email, aircraft upsert by name (findFirst then update/create, since name isn't unique). Demo users share password `password123`, stored only as bcrypt hash. Local setup order: db:setup -> db:push -> db:seed -> start.
- Models (prisma/schema.prisma): User(id,name,email unique,passwordHash,role,timestamps), Aircraft(id,name,model,status,capacity,timestamps), Booking(id,customerId FK,pickup/dest name+lat+lng,service,distanceKm,fareEstimate,status,operatorId? FK,aircraftId? FK,timestamps). Booking relations named "CustomerBookings"/"OperatorBookings" (two User FKs need named relations).
- BEFORE COMMIT: db-setup rewrites schema.prisma's provider to whatever .env says (sqlite locally). Restore the committed provider line to `mysql` before staging schema.prisma.
- DB engine is chosen by env: DATABASE_PROVIDER (mysql=default/Hostinger, sqlite=local) + DATABASE_URL. Prisma needs the provider as a literal, so `scripts/db-setup.js` rewrites the `provider = "..."` line in prisma/schema.prisma. ALWAYS run prisma via the npm scripts (`db:setup`/`db:push`/`db:migrate`) so the swap happens first. Committed schema default provider is `mysql`.
- Local dev workflow: set sqlite in .env -> `npm run db:setup` -> `npm run db:push` -> `npm start`. Verify with `curl localhost:3000/api/health`.
- Add new API routes to src/api.js (the `/api` express.Router). Use the shared Prisma client from `src/db.js` (`const { prisma } = require("./db")`).
- server.js connects to the DB on boot and `process.exit(1)`s with a FATAL message on failure (the "fail loudly" requirement).
- Typecheck = `npm run typecheck` (tsc --noEmit, checkJs:false, scoped to server.js/src/scripts via tsconfig.json — syntax-level check for this JS project). Keep it green.
- `.env`, `*.db`, `dev.db` are gitignored. `.env.example` documents both providers. node_modules gitignored.
- Do NOT commit the ralph harness dir (scripts/ralph/), .claude/, or tasks/ in feature commits — stage feature files explicitly.
- AUTH (src/auth.js): bcrypt password hashing + self-signed HMAC session tokens (no JWT dep, secret = AUTH_SECRET env). Endpoints under /api/auth (signup/login) return { token, user:{id,name,email,role} } — hash NEVER returned. Protect routes with `requireAuth` (sets req.user={id,name,role}) then `requireRole('operator'|'admin')` from src/auth.js. GET /api/me restores a session. Self-signups are always role "customer".
- TESTS: `npm test` = node's built-in runner (node --test) over test/**/*.js. Keep unit tests DB-free/hermetic (mock req/res, inject time into signToken/verifyToken). Add test/**/*.js to tsconfig include for any new test dir.
- FRONTEND: the whole customer/operator/admin app is ONE file, `app.html` (root) — inline `<style>` + one big inline `<script>` (the largest <script> block at the bottom). No build step, no modules; add functions to that block. Top-level screens are `<div class="view">`; exactly one has `.active`. Switch with `showView(id)` (clears .active on all `.view`, sets it on one). app.html is NOT in tsconfig, so `npm run typecheck` does NOT cover it — sanity-check inline JS with `node --check` on the extracted block. No browser MCP/dev-browser tool is configured in this env; verify client logic with a headless vm+stub harness hitting a live `node server.js`, and note that visual browser checks remain manual.
